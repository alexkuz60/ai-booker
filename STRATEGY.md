## Брифинг проекта: Ролевое AI веб-приложение для создания аудиокниги

### Цель проекта

Создание AI-Booker – веб-приложения для автоматизированного производства аудиокниг в формате радиоспектакля. Максимальная ИИ-автоматизация: ролевая озвучка персонажей с эмоциональной игрой, музыкально-шумовой фон, атмосфера места — при минимальном ручном вмешательстве.

---

### Ключевые идеи видения

1. **"От текста к иммерсивному спектаклю за один клик"** — упрощение создания сложных аудиопродуктов.
2. **"Human-Like" контент** — преодоление монотонности TTS: эмоциональная глубина, актёрская игра ИИ, паузы, атмосфера.
3. **Доступность для Creator Economy** — инди-авторы получают инструмент для качественных аудиоверсий без высоких затрат.
4. **Скорость разработки** — "вайб-кодинг" (Lovable.dev) для быстрой итерации и адаптации.

---

### Стратегические приоритеты

#### 1. Глубокий семантический разбор текста (LLM-Agent)
- Автоматическая структуризация (главы, сцены, диалоги)
- Идентификация персонажей (пол, возраст, психотип) → автоподбор голосов
- Эмоциональная разметка фраз → передача в движок озвучки
- Детекция триггеров для SFX и атмосферного фона

#### 2. Высококачественный синтез речи и клонирование голосов
- **Гибридная модель TTS**: Yandex SpeechKit / SaluteSpeech (русская фонетика) + ElevenLabs (эмоциональная выразительность) + ProxyAPI / OpenAI TTS (управляемая интонация через инструкции)
- **ProxyAPI TTS**: 3 модели (gpt-4o-mini-tts, tts-1, tts-1-hd), 9 базовых + 4 расширенных голоса, поддержка инструкций для эмоциональной игры
- **Смарт-разбивка**: автоматическая нарезка длинных блоков по границам предложений для обхода лимитов TTS API
- **RVC (Retrieval-based Voice Conversion)**: наложение уникальных тембров на базовую интонацию → "актёрская игра"
- SSML-разметка для тонкого контроля над речью (v1); overlay-синтез для v3-голосов без SSML

#### 3. The Atmosphere Engine — иммерсивный звуковой ландшафт
- Автогенерация контекстных SFX на основе текста
- Пространственная реверберация по описанной локации
- Динамический музыкальный/эмбиентный фон, адаптирующийся к настроению сцены

#### 4. Интуитивный интерфейс
- Мультитрековый таймлайн-редактор (Lovable.dev)
- "Regenerate" — повторная генерация фраз
- Редактор ударений

#### 5. Сведение и мастеринг (AudioEngine)
- Многоканальный аудио-движок на Tone.js с sample-accurate синхронизацией
- Персональный микшер на каждый трек: громкость, панорама, Mute/Solo, Pre-FX, Insert Reverb
- Мастер-цепочка: EQ3 → Compressor → Limiter → 5-Band Parametric Filter → 3-Band MBC → Reverb
- Спектроанализатор и стерео VU-метеринг в реальном времени
- Персистентность настроек микшера: localStorage (мгновенно) + cloud sync (user_settings, debounce 300ms)

### Character Profiler — автоматическое профилирование персонажей

> **Модуль NLP Core 1.2** — запускается после структурного разбора текста (The Architect). Анализирует диалоги и авторские ремарки, строит профили персонажей и рекомендует голоса.

#### Входные данные

- Разобранная структура книги (сцены, сегменты с типом `dialogue` / `narrator` / `inner_thought`)
- Текст глав/сцен с атрибуцией реплик (`speaker` в `scene_segments`)

#### Pipeline

| Шаг | Описание | Выход |
|-----|----------|-------|
| 1. **Извлечение** | LLM сканирует текст, собирает всех говорящих персонажей и их реплики | Список `{ name, aliases[], replica_count }` |
| 2. **Профилирование** | Для каждого персонажа LLM определяет характеристики на основе контекста | `{ gender, age_range, temperament, speech_style, language_register }` |
| 3. **Кластеризация** | Группировка псевдонимов и вариаций имён в единого персонажа | Нормализованный список уникальных персонажей |
| 4. **Voice Matching** | Подбор голоса из каталога (Yandex SpeechKit / ElevenLabs / ProxyAPI / пользовательские) по профилю с учётом psycho_tags, accentuation, archetype | `{ voice_id, confidence, alternatives[] }` |
| 5. **Превью** | Генерация короткого аудиосэмпла характерной реплики каждого персонажа (Yandex, ElevenLabs или ProxyAPI) | Аудио для подтверждения пользователем |
| 6. **Авто-кастинг** | Массовое назначение голосов всем персонажам без voice_config по профилю | Пол/возраст → голос, темперамент → роль, accentuation/archetype → TTS-пресет |

#### Профиль персонажа (структура данных)

```
Character {
  id: uuid
  book_id: uuid
  name: string                    // основное имя
  aliases: string[]               // «Иван Петрович», «Ваня», «старик»
  gender: "male" | "female" | "unknown"
  age_group: "child" | "teen" | "young" | "adult" | "elder" | "unknown"
  temperament: string             // «холерик», «меланхолик», etc.
  speech_style: string            // «грубый, отрывистый, с просторечиями»
  description: string             // краткий профиль от AI
  speech_tags: string[]           // теги манеры речи для TTS: #отрывисто #быстро #хрипло
  psycho_tags: string[]           // теги психотипа: #паникер #невротик #интроверт
  // Расширенные поля (в profile JSON / LocalCharacter):
  accentuation: string            // акцентуация по Леонгарду: гипертим/шизоид/истероид/эпилептоид/...
  archetype: string               // тембровый архетип: Мудрец/Герой/Опекун/Трикстер/...
  voice_config: json              // { provider, voice_id, role, speed, pitch, volume }
  color: string                   // цвет для UI-маркировки на таймлайне
  sort_order: number              // по частоте появления
}
```

#### Схема БД

**Таблица `book_characters`** — глобальный реестр персонажей книги:

| Поле | Тип | Описание |
|------|------|----------|

| `book_id` | uuid FK → books | Привязка к книге |

| `description` | text | Краткий профиль от AI |
| `speech_tags` | text[] | Теги манеры речи для TTS (#отрывисто, #быстро) |

### AI Roles — специализированные роли для LLM-задач ✅

> **Реализовано:** централизованный реестр AI-ролей с разделением по задачам. Каждая роль — отдельный system prompt, оптимальная модель и чёткая зона ответственности.

#### Реестр ролей

| Роль | Зона ответственности | Модель по умолчанию | Tier |
|------|----------------------|---------------------|------|
| **Переводчик** | Перевод запросов, i18n, локализация | gemini-2.5-flash-lite | lite |
| **Корректор** | Ударения, SSML, пунктуация TTS | gemini-2.5-flash | standard |
| **Сценарист** | **Парсер:** границы сцен (страницы + кол-во знаков). **Студия/Раскадровка:** разбивка на типизированные блоки (narrator, dialogue, monologue, inner_thought, first_person, lyric, epigraph, footnote, telephone, remark) + атрибуция спикеров | gemini-2.5-flash | standard |
| **Режиссёр** | Темп, паузы, BPM, эмоциональный рисунок глав | gemini-2.5-pro | heavy |
| **Профайлер** | Анализ персонажей, психотипы, стиль речи | gemini-2.5-pro | heavy |
| **Звукоинженер** | Промпты для SFX/атмосферы, звуковой дизайн | gemini-2.5-flash | standard |

#### Архитектура

- **Конфиг:** `src/config/aiRoles.ts` — определение ролей, system prompts, дефолтные модели
- **Хук:** `useAiRoles()` — маппинг роль→модель с user overrides через cloud settings
- **UI:** Профиль → таб «AI Роли» — переключение модели для каждой роли
- **Модели по tier:** lite (быстрые задачи), standard (баланс), heavy (глубокий анализ)
- **Принцип «Коней на переправе не меняют»:** модели фиксируются в профиле пользователя и применяются единообразно ко всему пайплайну (Парсер, Студия, Профайлер). Смена моделей после начала работы над книгой — осознанное действие. В будущем — поддержка вариантов аудиокниги для сравнения результатов разных моделей.

#### Будущее развитие

1. **Чаты пожеланий** — пользователь может дать указания каждому «специалисту» через отдельный чат
2. **Агентный pipeline** — роли работают последовательно или параллельно над сценой, передавая контекст
3. **Конфликт-резолвер** — если указания ролей противоречат, система запрашивает решение у пользователя
4. **Версионирование** — создание альтернативных вариантов аудиокниги с другим набором моделей для сравнения

---

### Универсальный стартовый набор (USN)

> **Вывод из опыта AI-Booker и СПРЗ Гидры:** перечисленные ниже модули являются фундаментом для любого многопользовательского вайб-проекта. Их следует закладывать **до** работы над доменной логикой — это экономит итерации и предотвращает рефакторинг на поздних этапах.

#### Категории применения в Гидре
Вайб-программирование · Технологии · Работа с файлами · Многопользовательский проект

#### Порядок реализации (рекомендуемый)

```
1. Layout + навигация + маршрутизация
2. Аутентификация + профиль + БД (profiles, user_roles)
3. Тема + язык + cloud settings (user_settings)
4. API-ключи + реестр моделей
5. Файлохранилище
6. API-роутеры + тестирование + логи
7. Админ-панель
8. → Доменная логика проекта
```

#### Принципы USN

- **Security-first**: RLS-политики на каждую таблицу, RBAC через security definer функции
- **Cloud-sync**: все пользовательские настройки — в БД, не только в localStorage
- **i18n с первого дня**: все строки через словарь, переключение без перезагрузки
- **Модульность**: каждый блок — отдельный компонент/хук, готовый к переносу между проектами
