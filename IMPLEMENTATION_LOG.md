# AI-Booker — Лог реализации СПРЗ

## Цель файла
Фиксация расхождений между планом СПРЗ (Гидра) и реальной реализацией для улучшения промптов Визионера, Стратега и Брифинга.

## 1. Аспект: Семантический анализ и Режиссура (NLP Core)

### 1.1. Модуль "The Architect" (Структурная декомпозиция)

| Параметр | План СПРЗ | Реализация | Расхождение |
|----------|-----------|------------|-------------|
| **Входной формат** | Не уточнён (подразумевался текст) | PDF (pdfjs-dist) | ✅ Уточнено в диалоге. СПРЗ не специфицировал формат файла — это пробел |
| **LLM для анализа** | GPT-5/Claude-Next | Гибрид: Lovable AI (gemini-3-flash-preview) по умолчанию + пользовательские ключи | ⚠️ СПРЗ указал конкретные модели, реализация гибкая — лучше чем план |
| **Очистка текста** | Промпт для LLM (сноски, номера страниц) | Реализовано в system prompt "The Architect" | ✅ Совпадает |
| **Сегментация** | Книга → Главы → Сцены | **Расширено:** Книга → Части → Главы → Сцены (4 уровня, DB: books → book_parts → book_chapters → book_scenes) | ⚠️ СПРЗ не предусмотрел уровень "Части" — выявлено при тестировании реальной книги |
| **TOC-first подход** | Не описан | PDF outline extraction → если нет TOC → LLM fallback | 🆕 Не предусмотрено СПРЗ. Идея пользователя. Кардинально улучшает точность и скорость |
| **Параллельный анализ** | Не описан | Главы анализируются параллельно (батчи по 3) с прогресс-баром | 🆕 Не предусмотрено СПРЗ. Идея пользователя. Решает проблему таймаутов |
| **Тип сцены** | action, dialogue, lyrical_digression | action, dialogue, lyrical_digression, description, inner_monologue, mixed | ✅ Расширено — больше типов чем в плане |
| **BPM** | Задание базового темпа | Реализовано (60-140+ диапазон в промпте) | ✅ Совпадает |
| **Настроение** | Определение настроения | Реализовано (tense, calm, melancholic и др.) | ✅ Совпадает |
| **Архитектура pipeline** | Не уточнена | Пошаговый с UI: Upload → TOC Review → Parallel Analysis → Results | ✅ Уточнено и расширено в диалоге |
| **Хранение результатов** | Не описано | Supabase: books + book_parts + book_chapters + book_scenes + RLS | ⚠️ СПРЗ не предусмотрел модель данных — пробел |
| **UI результатов** | Не описан | Интерактивное дерево: Части → Главы → Сцены с бейджами | ⚠️ СПРЗ не описал UI — пробел |
| **Пошаговый контроль** | Не описан | Шаг "Проверка структуры" — пользователь видит TOC перед запуском AI | 🆕 Реализация принципа "пошаговый с UI" |

#### Выводы по модулю 1.1 (обновлено)
- **Пробелы СПРЗ:** Формат файлов, модель данных, UI, TOC-подход, параллелизация, уровень "Части"
- **Сильные стороны СПРЗ:** Точное описание задач сегментации, типов сцен, BPM/mood
- **Что добавлено сверх плана:** TOC-first, параллельный анализ, 4 уровня иерархии, пошаговый UI с контролем
- **Ключевой вывод:** Тестирование на реальной книге выявило архитектурные пробелы, которые Стратег не мог предвидеть без знания формата данных
- **Рекомендации для промптов:** Стратег должен включать: Data Model, UI Flow, Input/Output Format, Fallback Strategy, Scalability (параллелизация)

### 1.2. Модуль "Character Profiler" (Кастинг-директор)

| Параметр | План СПРЗ | Реализация | Расхождение |
|----------|-----------|------------|-------------|
| **Хранение персонажей** | `Character { id, book_id, name, aliases[], ... }` | Таблица `book_characters` с RLS через JOIN на `books.user_id` | ✅ Совпадает, расширено полями `color`, `sort_order` |
| **Связь "персонаж ↔ сцена"** | Не описана в СПРЗ | Таблица `character_appearances` (character_id, scene_id, role_in_scene, segment_ids[]) | 🆕 Не предусмотрено СПРЗ — необходимо для графа появлений и будущего видеопроизводства |
| **Извлечение персонажей** | Отдельный LLM-проход по всей книге | **Инкрементально:** при сегментации сцены (`segment-scene`) speakers автоматически upsert в реестр | ⚠️ Гибридный подход: быстрое извлечение при сегментации + глубокий профайлинг позже |
| **Голосовые настройки** | `voice_id, voice_settings: json` | `voice_config: jsonb` — `{ provider, voice_id, role, speed, pitch, volume }` + ElevenLabs: `{ stability, similarity_boost, style, speed }` + ProxyAPI: `{ model, voice, instructions, speed }` | ✅ Расширено: мульти-провайдер (Yandex + ElevenLabs + ProxyAPI) с табовым UI |
| **UI персонажей** | Карточки с аватаром, Voice Audition, Bulk Assign | Список в Студии с настройкой голоса (табы Yandex/ElevenLabs/ProxyAPI), предпрослушивание, кредиты ElevenLabs, сохранение в БД | ⚠️ MVP-реализация; карточки, аватары и Bulk Assign — следующий этап |
| **Кластеризация алиасов** | LLM группирует псевдонимы | Реализовано: AI-профайлинг определяет алиасы, автоочистка дубликатов по совпадению имён/алиасов | ✅ Реализовано |
| **Voice Matching** | Автоподбор голоса по профилю | Авто-кастинг: пол/возраст → голос, темперамент → роль. Избегание дублирования голосов одного пола | ✅ Реализовано |
| **Системные персонажи** | Не описаны | «Рассказчик» и «Комментатор» — глобальные сущности книги с автопривязкой к типам блоков | 🆕 Не предусмотрено СПРЗ |
| **Консистентность** | Не описана | Новые спикеры при сегментации сопоставляются с существующими по имени/алиасам (регистронезависимо) | 🆕 Предотвращает дублирование |

#### Реализованные шаги

| Шаг | Описание | Статус | Детали |
|-----|----------|--------|--------|
| **1** | Таблицы `book_characters` + `character_appearances` с RLS | ✅ | Миграция, UNIQUE(book_id, name), каскадное удаление |
| **2** | Извлечение speakers при сегментации | ✅ | Edge Function `segment-scene` → upsert в `book_characters` + `character_appearances` |
| **3** | CharactersPanel с реестром из БД | ✅ | Загрузка по bookId, настройка голоса (табы Yandex/ElevenLabs/ProxyAPI), сохранение voice_config, кредиты ElevenLabs |
| **4** | AI-профайлинг | ✅ | `profile-characters` Edge Function: пол, возраст, темперамент, стиль, алиасы, инкрементальное обновление |
| **5** | Авто-кастинг голосов | ✅ | Автоподбор Yandex-голосов по профилю, автоназначение ролей по темпераменту |
| **6** | Автоочистка дубликатов | ✅ | Объединение по имени/алиасам с переносом character_appearances |

#### Архитектурные решения
- **Реестр голосов Yandex** вынесен в `src/config/yandexVoices.ts` (рефакторинг CharactersPanel)
- **Реестр голосов ElevenLabs** вынесен в `src/config/elevenlabsVoices.ts` (18 голосов, мультиязычные)
- **Реестр голосов ProxyAPI** вынесен в `src/config/proxyapiVoices.ts` (12 голосов: 9 базовых + 3 расширенных для gpt-4o-mini-tts; 3 модели с поддержкой инструкций)
- **Мульти-провайдер UI** — табы Yandex/ElevenLabs/ProxyAPI в панели персонажей, каждый со своими параметрами
- **Кредиты ElevenLabs** — Edge Function `elevenlabs-credits` + прогресс-бар в UI
- **Приоритет пользовательских ключей** — Edge Functions `elevenlabs-tts`/`elevenlabs-credits` сначала проверяют RPC `get_my_api_keys`, затем системный ключ
- **bookId** передаётся сквозь цепочку: Parser → sessionStorage → Studio → StudioWorkspace → CharactersPanel
- **Инкрементальный подход** выбран вместо полного сканирования: персонажи накапливаются по мере работы со сценами
- **Системные персонажи** (Рассказчик, Комментатор) — sort_order: 0, бейджи спикера скрыты в раскадровке
- **Синхронизация character_appearances** — при ручной смене/удалении спикера в раскадровке автоматически обновляется граф появлений

#### Выводы по модулю 1.2
- **Пробелы СПРЗ:** Не предусмотрена таблица связей "персонаж ↔ сцена", не описан инкрементальный подход, системные персонажи
- **Сильные стороны СПРЗ:** Точная структура профиля персонажа, pipeline из 5 шагов
- **Что добавлено сверх плана:** `character_appearances`, инкрементальное извлечение, multi-provider voice_config (Yandex + ElevenLabs + ProxyAPI), системные персонажи, автоочистка дубликатов, авто-кастинг, консистентность по алиасам

### 1.3. Модуль "Synthesis Pipeline" (Движок синтеза)

| Параметр | План СПРЗ | Реализация | Расхождение |
|----------|-----------|------------|-------------|
| **TTS-провайдер** | Yandex SpeechKit + ElevenLabs | Yandex SpeechKit (v1/v3), ElevenLabs (Multilingual v2), ProxyAPI/OpenAI TTS (gpt-4o-mini-tts, tts-1, tts-1-hd) — тройной API | ⚠️ Расширено: добавлен ProxyAPI с 3 моделями, 12 голосами и инструкциями |
| **Кэширование** | Не описано | `segment_audio` как кэш: FNV-1a хеш текста + voice_config → skip при совпадении | 🆕 Значительная оптимизация |
| **Ре-синтез** | Не описан | Индивидуальный (кнопка 🔄 на сегменте), ручная доозвучка (▶), пакетный по главе | 🆕 Трёхуровневая система |
| **Инлайн-вставки** | Не описаны | Двухпроходный синтез: SSML-паузы (v1) или overlay-треки (v3) | 🆕 Решает задачу авторских ремарок |
| **Выбор голоса вставок** | Не описан | Динамический: first_person speaker → или глобальный Рассказчик | 🆕 Акустическая консистентность |
| **Детекция устаревшего аудио** | Не описана | Сравнение voice_config в segment_audio с текущим voice_config персонажа; book-wide scan с разбивкой по провайдерам и предупреждением о расходах | 🆕 Иконка ⚠️ + батч ре-синтез по главе + батч по книге |
| **Плейлисты** | Не описаны | Таблица `scene_playlists`: полный снапшот (длительности, пути, спикеры) | 🆕 Оптимизация загрузки |
| **Yandex v3 ограничения** | Не описаны | Автосплит >240 символов по границам предложений, без SSML, валидация буфера | 🆕 Выявлено при тестировании |
| **Смарт-разбивка** | Не описана | splitTextForV3: автоматическая нарезка длинных текстов по предложениям → последовательный синтез → конкатенация буферов | 🆕 Обход лимита v3 |
| **Линковка first_person** | Не описана | scene_type_mappings: тип блока → персонаж; first_person определяет голос для вставок | 🆕 |
| **Слияние персонажей** | Не описано | Ручное и автоматическое (по совпадению имён/алиасов) с переносом appearances/segments | 🆕 |
| **ProxyAPI TTS** | Не описан | Edge Function `proxyapi-tts`: 3 модели, 12 голосов (9 базовых + 3 расширенных), инструкции для gpt-4o-mini-tts, валидация voice↔model совместимости | 🆕 Третий TTS-провайдер |

#### Выводы по модулю 1.3
- **Пробелы СПРЗ:** Весь синтез-пайплайн не был детализирован — только общая идея «TTS → аудио»
- **Что реализовано сверх плана:** Кэширование, ре-синтез, инлайн-вставки, устаревание аудио, плейлисты, v1/v3 совместимость, смарт-разбивка, линковка first_person, ProxyAPI TTS (3 модели, инструкции)

## 🚨 Критические контракты (НЕ МЕНЯТЬ без обоснования)

### К1. Диапазон страниц глав: resolvePageRange

**Контекст:** PDF outline часто содержит контейнерные узлы (например «Том 2» с диапазоном стр. 3–384, внутри которого «Акт 1» стр. 4–4). При наивном использовании `entry.startPage`–`entry.endPage` текст главы может оказаться пустым (1 страница = только заголовок).

**Контракт:**
- При анализе глав (`useChapterAnalysis.ts`) ВСЕГДА использовать `resolvePageRange(idx)` вместо прямого `entry.startPage`/`entry.endPage`
- `resolvePageRange` проверяет: подглавы (subtree), следующий sibling, и расширяет диапазон при необходимости
- Три уровня фоллбека: (1) subtree start–end, (2) расширение до nextSiblingStart-1, (3) повторная попытка с расширенным диапазоном при charCount < 50
- Лог в UI показывает `↔️ Уточнен диапазон` и `↪️ Текста мало, пробую...`

**Причина контракта:** Регресс 2026-03-14 — анализ возвращал «Text too short (min 50 chars)» для глав объёмом в сотни страниц. Лечение симптома (снижение порога до 10) не решало проблему.

### К2. Импорт TOC: контейнерные узлы пропускаются

**Контекст:** PDF outline может содержать иерархические узлы-контейнеры (например «Том 2» содержит дочерние «Акт 1», «Акт 2»…). Контейнеры не содержат собственного текста — только заголовок.

**Контракт:**
- При импорте TOC из PDF outline (`useBookManager.ts`, `handleFileSelect`) узлы с `children.length > 0` НЕ добавляются как главы
- Вместо этого: контейнеры level=0 запоминаются как `currentPart` (название части), остальные контейнеры просто пропускаются
- Фоллбек: если после фильтрации контейнеров `chapters.length === 0`, берём ВСЕ узлы (плоский список)

**Причина контракта:** Регресс 2026-03-14 — контейнерные узлы создавали главы с диапазоном 1 страница, что приводило к пустым сценам и ошибкам анализа.

### Р1. Главы без текста: контейнерные узлы TOC + некорректный resolvePageRange

**Дата:** 2026-03-14  
**Симптомы:**
- Анализ глав возвращал "Text too short (min 50 chars)" для глав объёмом в сотни страниц
- Главы создавались с диапазоном 1 страница (например, 4-4 вместо 4-67)
- Пользователь получал пустые сцены и ошибки сегментации

**Причина:**
1. **Импорт контейнеров TOC:** `useBookManager.ts` добавлял в БД узлы-контейнеры (с `children.length > 0`), которые содержат только заголовок без текста
2. **Некорректное использование page range:** `useChapterAnalysis.ts` напрямую использовал `entry.startPage`/`entry.endPage` вместо `resolvePageRange(idx)`, что не расширяло диапазон для вложенных структур

**Лечение симптома (НЕПРАВИЛЬНО):**
- Снижение порога `minLen` с 50 до 10 символов — маскировало проблему, но не решало
- Попытки модифицировать промпт LLM для работы с короткими текстами

**Правильное решение:**
1. Зафиксирован контракт **К1** (resolvePageRange)
2. Зафиксирован контракт **К2** (пропуск контейнеров при импорте)
3. Обновлён `useChapterAnalysis.ts`: всегда использовать `resolvePageRange(idx)`
4. Обновлён `useBookManager.ts`: фильтровать узлы с `children.length > 0`

**Паттерн:** Лечение симптомов вместо поиска root cause. AI пыталась исправить пороги и промпты, вместо проверки данных в БД.

**Профилактика:** 
- Всегда проверять фактические данные (диапазон страниц в БД, длину извлечённого текста) перед изменением логики
- Критические архитектурные решения фиксировать в контрактах
- При повторении ошибки "слишком короткий текст" — проверить `resolvePageRange`, а не порог `minLen`

- AI повторно вносит изменения, которые противоречат ранее зафиксированным контрактам
- Убеждает пользователя, что "новое решение" лучше, несмотря на наличие документации
- Не проверяет IMPLEMENTATION_LOG.md и STRATEGY.md перед изменениями

**Причина:**
- Отсутствие cross-session памяти у AI
- Документация существует, но не читается проактивно
- Новые запросы обрабатываются в изоляции от исторического контекста

**Правильное решение:**
1. Чек-лист обязательных проверок добавлен в начало STRATEGY.md
2. Критические контракты зафиксированы в IMPLEMENTATION_LOG.md (раздел "🚨 Критические контракты")
3. Этот раздел "Известные регрессии" создан для отслеживания паттернов

**Паттерн:** "Амнезия" между сессиями приводит к циклическим регрессиям и потере времени.

**Профилактика:**
- **Перед любым изменением кода:** читать STRATEGY.md (чек-лист), IMPLEMENTATION_LOG.md (контракты), WORKFLOW.md (контекст)
- При запросе на "улучшение" проверить, не было ли это уже реализовано и зафиксировано
- В случае сомнений — спросить пользователя, а не действовать автономно
- Рассмотреть автоматизацию: git pre-commit hooks для проверки обновления документации при изменении критических файлов
