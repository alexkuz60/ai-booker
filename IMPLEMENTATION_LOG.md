# AI-Booker — Лог реализации СПРЗ

## Цель файла
Фиксация расхождений между планом СПРЗ (Гидра) и реальной реализацией для улучшения промптов Визионера, Стратега и Брифинга.

## 1. Аспект: Семантический анализ и Режиссура (NLP Core)

### 1.1. Модуль "The Architect" (Структурная декомпозиция)

| Параметр | План СПРЗ | Реализация | Расхождение |
|----------|-----------|------------|-------------|
| **Входной формат** | Не уточнён (подразумевался текст) | PDF (pdfjs-dist) | ✅ Уточнено в диалоге. СПРЗ не специфицировал формат файла — это пробел |
| **LLM для анализа** | GPT-5/Claude-Next | Гибрид: Lovable AI (gemini-3-flash-preview) по умолчанию + пользовательские ключи | ⚠️ СПРЗ указал конкретные модели, реализация гибкая — лучше чем план |
| **Очистка текста** | Промпт для LLM (сноски, номера страниц) | Реализовано в system prompt "The Architect" | ✅ Совпадает |
| **Сегментация** | Книга → Главы → Сцены | **Расширено:** Книга → Части → Главы → Сцены (4 уровня, DB: books → book_parts → book_chapters → book_scenes) | ⚠️ СПРЗ не предусмотрел уровень "Части" — выявлено при тестировании реальной книги |
| **TOC-first подход** | Не описан | PDF outline extraction → если нет TOC → LLM fallback | 🆕 Не предусмотрено СПРЗ. Идея пользователя. Кардинально улучшает точность и скорость |
| **Параллельный анализ** | Не описан | Главы анализируются параллельно (батчи по 3) с прогресс-баром | 🆕 Не предусмотрено СПРЗ. Идея пользователя. Решает проблему таймаутов |
| **Тип сцены** | action, dialogue, lyrical_digression | action, dialogue, lyrical_digression, description, inner_monologue, mixed | ✅ Расширено — больше типов чем в плане |
| **BPM** | Задание базового темпа | Реализовано (60-140+ диапазон в промпте) | ✅ Совпадает |
| **Настроение** | Определение настроения | Реализовано (tense, calm, melancholic и др.) | ✅ Совпадает |
| **Архитектура pipeline** | Не уточнена | Пошаговый с UI: Upload → TOC Review → Parallel Analysis → Results | ✅ Уточнено и расширено в диалоге |
| **Хранение результатов** | Не описано | Supabase: books + book_parts + book_chapters + book_scenes + RLS | ⚠️ СПРЗ не предусмотрел модель данных — пробел |
| **UI результатов** | Не описан | Интерактивное дерево: Части → Главы → Сцены с бейджами | ⚠️ СПРЗ не описал UI — пробел |
| **Пошаговый контроль** | Не описан | Шаг "Проверка структуры" — пользователь видит TOC перед запуском AI | 🆕 Реализация принципа "пошаговый с UI" |

#### Выводы по модулю 1.1 (обновлено)
- **Пробелы СПРЗ:** Формат файлов, модель данных, UI, TOC-подход, параллелизация, уровень "Части"
- **Сильные стороны СПРЗ:** Точное описание задач сегментации, типов сцен, BPM/mood
- **Что добавлено сверх плана:** TOC-first, параллельный анализ, 4 уровня иерархии, пошаговый UI с контролем
- **Ключевой вывод:** Тестирование на реальной книге выявило архитектурные пробелы, которые Стратег не мог предвидеть без знания формата данных
- **Рекомендации для промптов:** Стратег должен включать: Data Model, UI Flow, Input/Output Format, Fallback Strategy, Scalability (параллелизация)

### 1.2. Модуль "Character Profiler" (Кастинг-директор)

| Параметр | План СПРЗ | Реализация | Расхождение |
|----------|-----------|------------|-------------|
| **Хранение персонажей** | `Character { id, book_id, name, aliases[], ... }` | Таблица `book_characters` с RLS через JOIN на `books.user_id` | ✅ Совпадает, расширено полями `color`, `sort_order` |
| **Связь "персонаж ↔ сцена"** | Не описана в СПРЗ | Таблица `character_appearances` (character_id, scene_id, role_in_scene, segment_ids[]) | 🆕 Не предусмотрено СПРЗ — необходимо для графа появлений и будущего видеопроизводства |
| **Извлечение персонажей** | Отдельный LLM-проход по всей книге | **Инкрементально:** при сегментации сцены (`segment-scene`) speakers автоматически upsert в реестр | ⚠️ Гибридный подход: быстрое извлечение при сегментации + глубокий профайлинг позже |
| **Голосовые настройки** | `voice_id, voice_settings: json` | `voice_config: jsonb` — `{ provider, voice_id, role, speed, pitch, volume }` + ElevenLabs: `{ stability, similarity_boost, style, speed }` | ✅ Расширено: мульти-провайдер (Yandex + ElevenLabs) с табовым UI |
| **UI персонажей** | Карточки с аватаром, Voice Audition, Bulk Assign | Список в Студии с настройкой голоса (табы Yandex/ElevenLabs), предпрослушивание, кредиты ElevenLabs, сохранение в БД | ⚠️ MVP-реализация; карточки, аватары и Bulk Assign — следующий этап |
| **Кластеризация алиасов** | LLM группирует псевдонимы | Поле `aliases text[]` готово, кластеризация — на Шаге 4 | ⏳ Ожидает глубокого профайлинга |
| **Voice Matching** | Автоподбор голоса по профилю | Ручной выбор из каталога Yandex SpeechKit | ⏳ Автоматический подбор — следующий этап |

#### Реализованные шаги

| Шаг | Описание | Статус | Детали |
|-----|----------|--------|--------|
| **1** | Таблицы `book_characters` + `character_appearances` с RLS | ✅ | Миграция, UNIQUE(book_id, name), каскадное удаление |
| **2** | Извлечение speakers при сегментации | ✅ | Edge Function `segment-scene` → upsert в `book_characters` + `character_appearances` |
| **3** | CharactersPanel с реестром из БД | ✅ | Загрузка по bookId, настройка голоса (табы Yandex/ElevenLabs), сохранение voice_config, кредиты ElevenLabs, индикатор привязки |
| **4** | Глубокий AI-профайлинг | ⏳ | description, temperament, speech_style, кластеризация алиасов |

#### Архитектурные решения
- **Реестр голосов Yandex** вынесен в `src/config/yandexVoices.ts` (рефакторинг CharactersPanel)
- **Реестр голосов ElevenLabs** вынесен в `src/config/elevenlabsVoices.ts` (18 голосов, мультиязычные)
- **Мульти-провайдер UI** — табы Yandex/ElevenLabs в панели персонажей, каждый со своими параметрами
- **Кредиты ElevenLabs** — Edge Function `elevenlabs-credits` + прогресс-бар в UI
- **Приоритет пользовательских ключей** — Edge Functions `elevenlabs-tts`/`elevenlabs-credits` сначала проверяют RPC `get_my_api_keys`, затем системный ключ
- **bookId** передаётся сквозь цепочку: Parser → sessionStorage → Studio → StudioWorkspace → CharactersPanel
- **Инкрементальный подход** выбран вместо полного сканирования: персонажи накапливаются по мере работы со сценами, что даёт мгновенный результат без задержки на анализ всей книги

#### Выводы по модулю 1.2
- **Пробелы СПРЗ:** Не предусмотрена таблица связей "персонаж ↔ сцена", не описан инкрементальный подход к извлечению
- **Сильные стороны СПРЗ:** Точная структура профиля персонажа, pipeline из 5 шагов
- **Что добавлено сверх плана:** `character_appearances`, инкрементальное извлечение, multi-provider voice_config
- **Стратегическая перспектива:** Реестр + граф появлений — фундамент для видеопроизводства (casting sheet)

## Общие наблюдения

### Что не предусмотрено Стратегом
1. Формат входных файлов (PDF/TXT/EPUB)
2. Модель данных (таблицы, связи, RLS)
3. UI/UX flow (шаги, визуализация результатов)
4. Выбор LLM провайдера и fallback-стратегия
5. Аутентификация и доступ к данным
6. **USN как обязательный этап** — не был выделен в отдельную фазу

### Что оказалось лишним
- Пока нет лишних элементов — все задачи релевантны

### Рекомендации по улучшению промптов
- **Визионер:** Добавить требование описывать user journey для каждого модуля
- **Стратег:** Включить обязательные секции: Data Model, UI Flow, Error Handling, Provider Strategy
- **Брифинг:** Указывать конкретные форматы ввода/вывода для каждого модуля
- **Гидра:** Включить USN-чеклист как первый этап плана для всех категорий с бэкендом
