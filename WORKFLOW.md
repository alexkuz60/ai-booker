# AI-Booker: Рабочий процесс создания аудиокниги

> Документ описывает полный пайплайн — от загрузки файла до финального рендера.  
> Актуальная дата: 2026-03-21. Статус каждого шага: ✅ реализовано, 🔧 частично, ⬜ запланировано.

---

## 1. Загрузка книги ✅

**Модуль:** Парсер → Библиотека → Загрузка  
**Хуки:** `useFileUpload`, `useProjectStorage`

| Параметр | Детали |
|----------|--------|
| Форматы файлов | **PDF** (`pdfjs-dist`), **DOCX** (`mammoth.js`), **FB2** (нативный XML-парсер) |
| Хранение файлов | **Только локально** в OPFS/FS Access (`source/book.{ext}`). На сервер файл НЕ загружается |
| Создание проекта | Диалог ввода имени → создание именованной директории → `project.json` + структура папок |
| Ограничения | Обработка целиком в браузере, размер не ограничен |

**Порядок действий пользователя:**
1. Открыть модуль **Парсер**.
2. Ввести имя проекта и выбрать файл (PDF/DOCX/FB2).
3. Автоматическое извлечение оглавления и создание локального проекта.

## 2. Извлечение структуры (TOC) ✅

**Модуль:** Парсер (автоматический этап)  
**Хуки:** `useFileUpload` → `mapFlatToChapters()`

Сразу после загрузки файла система:

### PDF:
1. Извлекает встроенное оглавление (`pdf.getOutline()`) — иерархически, с определением уровней вложенности.
2. Вычисляет диапазоны страниц для каждой записи.
3. Fallback: если outline пуст → извлечение TOC из текста первых страниц.

### DOCX:
1. Извлекает заголовки из Heading-стилей (H1-H6) через `mammoth.js`.
2. Regex-фоллбэк для документов без стилей.
3. Контент кэшируется в module-level `Map` (`chapterTextsCache.ts`) для последующего анализа. ⚠️ Не в sessionStorage (контракт К4).

### FB2:
1. Парсит XML-структуру `<section>/<title>`.
2. Автоматически определяет уровни вложенности.

### Общее для всех форматов:
- Классификация по типу раздела: содержание, предисловие, послесловие, примечания, приложения.
- Нормализация уровней вложенности (устраняет «осиротевшие» подпункты).
- **Все ID генерируются на клиенте** (`crypto.randomUUID()`). Запись в БД НЕ выполняется.
- Структура сохраняется в OPFS: `structure/toc.json` + `structure/chapters.json`.

## 3. Навигатор структуры книги ✅

**Модуль:** Парсер → Рабочая область (левая панель)  
**Компонент:** `NavSidebar`  
**Хуки:** `useTocMutations`

Навигатор отображает полную структуру книги в виде дерева с возможностью ручного редактирования:

### Отображение
- Древовидная структура с группировкой по частям (Part I, Part II…)
- Раскрываемые узлы для частей и глав с подглавами
- Разделение на секции: содержание, предисловие, послесловие, примечания, приложения
- Индикаторы статуса каждой главы: ⏳ ожидает / ⚙️ анализ / ✅ готово / ❌ ошибка
- Ссылки на страницы PDF (клик открывает PDF на нужной странице)

### Инструменты редактирования
| Инструмент | Описание |
|------------|----------|
| **Inline-редактирование заголовка** | Двойной клик по заголовку главы → ввод нового названия |
| **Редактирование начальной страницы** | Двойной клик по номеру страницы → изменение (автоматически корректируется конечная страница предыдущей главы) |
| **Изменение уровня вложенности** | Кнопки `<` (outdent) и `>` (indent) — рекурсивно смещают дочерние элементы |
| **Удаление записей** | Кнопка 🗑️ — удаляет запись и все дочерние элементы (с подтверждением) |
| **Переименование частей** | Двойной клик по заголовку части |
| **Открытие PDF** | Кнопка 🔗 — открывает PDF в новой вкладке на нужной странице |
| **Мультивыбор** | `Ctrl+Click` (добавить/убрать), `Shift+Click` (диапазон) — массовые операции |

Все изменения **мгновенно сохраняются в OPFS** через `useImperativeSave`. Запись в БД — только по кнопке «На сервер».

## 4. Семантический анализ глав ✅

**Модуль:** Парсер → Рабочая область (правая панель) + Edge Function `parse-book-structure`  
**Хуки:** `useChapterAnalysis`

### Процесс (двухэтапный)

**Этап 1 — Определение границ сцен:**
1. Извлечение текста из файла по диапазону страниц главы (PDF) или из sessionStorage-кэша (DOCX/FB2).
2. Отправка текста в AI с режимом `boundaries` — определение начальных строк каждой сцены.
3. Разделение текста по маркерам на отдельные сцены.

**Этап 2 — Обогащение метаданными:**
1. Для каждой сцены AI определяет (режим `enrich`):
   - **Тип сцены**: action, dialogue, lyrical_digression, description, inner_monologue, mixed
   - **Настроение** (mood): тональная характеристика
   - **BPM**: темп сцены (числовое значение)
   - **Заголовок**: краткое название сцены
2. Результаты сохраняются **только в OPFS** (`scenes/chapter_{id}.json`).

### Каскадный fallback провайдеров
При ошибках 402/429 система автоматически переключается между провайдерами:
- Lovable AI → OpenRouter → ProxyAPI → DotPoint
- Переключение прозрачно для пользователя (toast-уведомление)
- Реализовано через `invokeWithFallback` + `_shared/providerRouting.ts`

### Выбор AI-модели
Модель определяется через AI-роль «Сценарист» в профиле пользователя:
- **Встроенные** (без ключа, только админы): Gemini 3 Flash/Pro, GPT-5/Mini/Nano
- **ProxyAPI** (нужен ключ): Claude 4 Sonnet/Opus, GPT-5/5-mini
- **OpenRouter** (нужен ключ): любая модель из каталога

### UX анализа
- Прогресс-лог в реальном времени (определение границ → расчёт страниц → настроение/темп для каждой сцены)
- Фоновый предвыбор (prefetch) до 3 следующих глав
- Возможность повторного/продолжения анализа
- Карточки сцен с цветовой кодировкой по типу, длительностью и раскрываемым превью текста

## 5. Передача главы в Студию ✅

**Модуль:** Парсер → Студия

### Условия доступности
Кнопка переноса (🎬) отображается когда:
- Глава верхнего уровня: сама глава и **все** её дочерние подглавы имеют статус `done` с непустыми сценами.
- Подглава: аналогичная рекурсивная проверка.

### Механизм передачи
1. Собираются сцены из самой главы и всех её дочерних подглав.
2. Для вложенных элементов заголовок формируется как: `«Название родителя (Часть N)»`, где N — порядковый номер.
3. **Метаданные (указатели)** сохраняются в `sessionStorage`: bookId, chapterId, chapterTitle, bookTitle, selectedSceneIdx, activeTab. ⚠️ Текст сцен НЕ кладётся в sessionStorage (контракт К4).
4. Пользователь перенаправляется на страницу Студии.
5. `useStudioSession` восстанавливает главу из OPFS по сохранённым указателям.

## 6. Студия: навигатор сцен ✅

**Модуль:** Студия (левая панель)

- Отображение списка сцен загруженной главы.
- Цветовая кодировка по типу сцены (action, dialogue, description…).
- Оценка длительности каждой сцены и главы в целом (по количеству символов).
- Статистика в хедере: общая длительность главы, количество символов.

### Индикаторы статуса аудио ✅
Каждая сцена в навигаторе отображает иконку готовности:
- 🎬 **Film** — нет аудио (только раскадровка).
- 🔈 **Контурный динамик** — частично отрендерено (есть аудио, но не для всех сегментов).
- 🔊 **Сплошной динамик** — полностью синтезирована (все сегменты имеют аудио).
- ⚠️ **Жёлтый треугольник** — аудио устарело (параметры голоса персонажа изменены после синтеза).

### Пакетный ре-синтез ✅
Если хотя бы одна сцена помечена как «устаревшая» (⚠️), в хедере навигатора появляется кнопка пакетного ре-синтеза. При нажатии:
1. Автоматически находятся все сцены с устаревшим аудио.
2. Последовательно запускается ре-синтез каждой сцены.
3. Прогресс отображается в реальном времени.

### Сканирование всей книги на устаревшее аудио ✅
Кнопка **📖** (BookOpen) в хедере навигатора запускает сканирование **всех глав книги** на предмет расхождения голосовых настроек:
1. Голосовые конфигурации персонажей читаются из `characters/index.json` (OPFS, контракт K4). Fallback на `book_characters` (DB) — только при отсутствии локальных данных.
2. Загружаются все `segment_audio` для всех сцен книги.
3. Для каждого сегмента сравниваются 7 ключевых параметров (`voice`, `role`, `speed`, `pitchShift`, `volume`, `provider`, `model`) текущей конфигурации персонажа с сохранённой при синтезе.
4. Результат отображается в **AlertDialog** с:
   - Общим количеством устаревших сегментов.
   - Числом затронутых сцен.
   - Разбивкой по провайдерам (ElevenLabs / Yandex / ProxyAPI) для оценки стоимости.
   - Предупреждением о дополнительных расходах.
5. При подтверждении — массовый ре-синтез только изменившихся сегментов (через `segment_ids`), сцена за сценой с отображением прогресса.

## 7. Раскадровка (Storyboard) ✅

**Модуль:** Студия → вкладка «Раскадровка» + Edge Function `segment-scene`

### Автоматическая сегментация
При выборе сцены в навигаторе система автоматически:
1. Проверяет, есть ли уже сегменты в БД для данной сцены.
2. Если нет — запускает AI-анализ, разбивающий текст сцены на типизированные сегменты.

### Пост-обработка: детекция повествования от первого лица ✅
После AI-сегментации применяется regex-детекция: блоки, размеченные как `narrator`, но содержащие местоимения первого лица (я, мне, меня, мной, мною, моего, моей, моему, моим, моими, моих, моё, мое, мои / I, me, my, mine, myself), автоматически переклассифицируются в `first_person`. Обеспечивает корректный подбор голоса для повествования от первого лица.

### Типы сегментов
| Тип | Описание |
|-----|----------|
| 📖 **Рассказчик** (narrator) | Повествование от третьего лица |
| 👤 **От первого лица** (first_person) | Повествование от первого лица |
| 💬 **Диалог** (dialogue) | Прямая речь персонажа (с указанием спикера) |
| 🧠 **Мысли** (inner_thought) | Внутренний монолог |
| ✍️ **Эпиграф** (epigraph) | Цитата или эпиграф |
| 🎵 **Лирика** (lyric) | Стихотворный или песенный фрагмент |
| 📝 **Сноска** (footnote) | Примечание |

### Структура данных
Каждый сегмент разбивается на **фразы** (предложения) — атомарные единицы для озвучки:
- `scene_segments` → тип, спикер, порядковый номер, metadata (jsonb)
- `segment_phrases` → текст фразы, порядковый номер

### Редактирование
- Каждая фраза редактируемая inline (клик по иконке ✏️ → textarea → Enter для сохранения, Esc для отмены).
- Изменения сохраняются в БД мгновенно.
- Возможность переанализа сцены (кнопка «Переанализ»).

### Интерактивные бейджи сегментов ✅
- **Тип блока** (narrator, dialogue, first_person…) — кликабельный бейдж-селектор. Позволяет вручную исправить ошибку классификации AI. Выбор сохраняется в БД.
- **Персонаж** — кликабельный бейдж-селектор. Если спикер не назначен, отображается «персонаж ?» (оранжевый). Выбор доступен из списка персонажей книги.
- **Связка тип ↔ персонаж**: при назначении персонажа на блок определённого типа (narrator, first_person, inner_thought, epigraph, lyric, footnote) — все остальные блоки того же типа в пределах сцены автоматически получают того же персонажа. Для **dialogue** смена применяется строго индивидуально.
- **Пропагация типов narrator↔first_person**: при смене типа одного блока с `narrator` на `first_person` (или обратно) — **все** блоки старого типа в сцене автоматически меняются на новый. После пропагации запускается `syncSceneCharacters`, удаляющий персонажей без оставшихся сегментов → пустые треки на таймлайне исчезают.
- Связка работает в пределах одной сцены и синхронизирована с табом «Персонажи» и таймлайном.

### Детекция инлайн-вставок (Inline Narrations) ✅

**Edge Function:** `detect-inline-narrations`

В русской литературе внутри прямой речи часто встречаются авторские ремарки:
> «Родя, — *тихо позвал он,* — ты не умирай»

Кнопка **«Вставки»** (иконка ScanSearch) в тулбаре Раскадровки запускает AI-детекцию:

1. Загружает все диалоговые сегменты сцены.
2. AI (Gemini 2.5 Flash) анализирует каждый сегмент и выделяет:
   - **inline_narrations** — массив `{ text, insert_after }` (текст вставки и позиция).
   - **clean_text** — текст диалога с удалёнными авторскими словами.
3. Найденные вставки сохраняются в `metadata.inline_narrations` сегмента.
4. Оригинальные фразы очищаются от авторского текста (предотвращает дублирование при синтезе).

**Отображение:**
- В заголовке сегмента — иконка 💬 с количеством вставок.
- Под текстом фраз — бледно-жёлтый блок в формате: `после [фраза] → [вставка]`.

### Индикация аудио в сегментах ✅
Каждый сегмент отображает статус аудио и кнопку управления:
- ✅ **Готово** — зелёная галочка + длительность (`2.5s`).
- ▶ **Нет аудио** — кнопка ручного синтеза отдельного сегмента.
- 🔄 **Ре-синтез** — кнопка принудительного обновления существующего аудио.
- ❌ **Ошибка** — красный крестик.

### Синхронизация с другими модулями ✅
- При смене/назначении спикера — обновляется `characters/scene_{id}.json` (OPFS, контракт K4) → дорожка на таймлайне.
- При смене типа блока `narrator↔first_person` все блоки того же типа в сцене автоматически обновляются (пропагация).
- Удаление сегмента → scene character map пересчитывается → треки без сегментов удаляются.
- Изменения фраз немедленно отражаются в хэш-кэше синтеза.

## 8–14. Студия: Таймлайн, Персонажи, TTS, AudioEngine, Рендеринг

> Подробное описание этих шагов остаётся прежним. Ключевое изменение: все данные хранятся и читаются из OPFS (контракты К3, К4). Персонажи, голосовые конфигурации и связи «спикер↔сегмент» — из `characters/index.json` и `characters/scene_{id}.json`. Переход из Парсера в Студию — через указатели в sessionStorage + `useStudioSession` для восстановления контента из OPFS.

### Ключевые реализованные модули:
- ✅ Мультитрековый таймлайн с воспроизведением и seek
- ✅ Персонажи: извлечение, профайлинг, авто-кастинг, мульти-провайдер голоса
- ✅ TTS: Yandex SpeechKit (v1/v3) + ElevenLabs + ProxyAPI/OpenAI (3 модели)
- ✅ AudioEngine (Tone.js): треки, микшер, мастер-цепочка (EQ → Comp → Lim → Filter → MBC → Reverb)
- ✅ Per-Clip плагины: EQ, Compressor, Limiter, Panner3D, Convolver
- ✅ Рендеринг сцены в стемы (Voice/Atmo/SFX → WAV)
- ✅ Atmosphere Engine: ручная/авто генерация, Freesound, DB-хранение

## 15. Библиотека проектов ✅

**Модуль:** Парсер → Библиотека  
**Хуки:** `useLibrary`

### Источник данных
- **Только OPFS** — библиотека отображает проекты из локального хранилища.
- Каждый проект = папка в OPFS с `project.json` (метаданные).
- Fallback на `structure/toc.json` при повреждении `project.json`.

### Функции
- Список проектов с именем, датой создания, статусом
- Открытие проекта → восстановление state из OPFS
- Удаление проекта (с подтверждением)
- «Очистить всё» — пакетное удаление всех проектов
- Аварийный сброс: `?resetLocal=1` → полная очистка OPFS и кэшей

### Серверная синхронизация
- При открытии книги: проверка `books.updated_at` vs `project.json.updatedAt`
- Диалог подтверждения при наличии более новой серверной версии
- ⬜ **Планируется:** кнопка «Загрузить с сервера» для New Workstation Flow
