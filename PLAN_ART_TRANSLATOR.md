# Art Translator — План модуля художественного перевода

> Статус: **F8 завершён — батчинг + пул моделей** | Обновлено: 2026-03-28

## 1. Цель

Многоагентный пайплайн профессионального художественного перевода книги (RU→EN и обратно),
с объективной оценкой качества через многовекторное сравнение эмбеддингов —
**без необходимости знания целевого языка пользователем**.

## 2. Ключевые принципы

- **Ролевой подход** (паттерн «AI-Гидра»): каждый агент — специалист с чёткой зоной ответственности.
- **Фонетические эмбеддинги**: программная реализация для точного сравнения просодии и ритмики.
- **Измеримость без билингвальности**: радар-диаграмма эмбеддингов позволяет оценить качество перевода
  по объективным метрикам (семантика, тональность, ритмика, культурный код).
- **Сохранение метаданных**: SSML-разметка, BPM, психотипы персонажей, паузы — переносятся в перевод
  для бесшовной TTS-интеграции на целевом языке.
- **Сегменты Студии как вход**: раскадровка (storyboard) оригинала — готовая «партитура» с типами,
  спикерами и аннотациями, структурирующая задание для каждого агента.

## 3. Архитектура: Dual-Project

### 3.1. Принцип

Перевод хранится как **отдельный OPFS-проект** с идентичной иерархией ID.

```
МояКнига/                          ← оригинал
  project.json                     ← { language: "ru" }
  scene_index.json
  characters.json
  chapters/{chId}/scenes/{scId}/
    storyboard.json

МояКнига_EN/                       ← перевод
  project.json                     ← { language: "en",
                                        sourceProjectName: "МояКнига",
                                        targetLanguage: "en" }
  scene_index.json                 ← копия структуры оригинала
  characters.json                  ← переведённые имена/описания
  chapters/{chId}/scenes/{scId}/
    storyboard.json                ← переведённые фразы
    radar.json                     ← оценки качества по осям (per-segment)
```

### 3.2. Преимущества

- Все ID глав/сцен/сегментов **идентичны** → бесплатный маппинг «оригинал ↔ перевод».
- `ProjectStorage` переиспользуется as-is, `paths.*` работают без изменений.
- Push to Server — независимый для каждой версии.
- Оригинал остаётся **read-only** в контексте страницы перевода.

### 3.3. ProjectMeta (расширение)

```typescript
interface ProjectMeta {
  // ... existing fields ...
  sourceProjectName?: string;   // OPFS-имя проекта оригинала (только у переводов)
  targetLanguage?: "en" | "ru"; // целевой язык (только у переводов)
}
```

### 3.4. Создание проекта-перевода

1. Пользователь нажимает «Создать перевод» на странице `/translation`.
2. Копируется структура: `toc.json`, `scene_index.json`, `chapters/` (с пустыми storyboard).
3. `project.json` заполняется `sourceProjectName` + `targetLanguage`.
4. Персонажи копируются для последующего перевода имён/описаний.

## 4. Агентный пайплайн

### 4.1. Роли

| Роль | ID | Tier | Задача |
|------|----|------|--------|
| **Арт-переводчик** (Art Translator) | `art_translator` | standard | Дословный + литературный перевод сегментов |
| **Литературный редактор** (Literary Editor) | `literary_editor` | heavy | Стилистическая доводка, ритмика, культурная адаптация |
| **Критик перевода** (Translation Critic) | `translation_critic` | standard | Оценка качества по 5 осям Quality Radar |

> Системный `translator` (lite) остаётся для утилитарных переводов (поиск, промпты SFX/музыки).

### 4.2. Цикл обработки (на уровне сегмента)

Сегмент — **атомарная единица** перевода. Внутри одной сцены сегменты различаются по типу
(повествование, диалог, мысли, стихи), спикеру, настроению и BPM. Эмбеддинги и оценка
качества вычисляются именно на этом уровне — потому что смысл, интонация и ритм меняются
от сегмента к сегменту.

UI: каждый сегмент отображается как аккордеон-блок. Внутри — **оригинал** (read-only)
и **перевод** (редактируемый) друг под другом.

```
Оригинал (сегмент из storyboard)
  │
  ├─→ [Переводчик] → подстрочник (translate-literal EF)
  │
  ├─→ [Лит. редактор] → художественный перевод (translate-literary EF)
  │       (получает: подстрочник + оригинал + контекст персонажа + SSML/BPM)
  │
  ├─→ [Программный аудит] → ритмика + фонетика (phoneticFeatures.ts, мгновенно)
  │
  ├─→ [Семантический аудит] → cosine similarity (embeddingClient.ts, API)
  │
  ├─→ [Критик] → тональность + культурный код + рекомендации (critique-translation EF)
  │       (получает: оригинал + перевод + дельты эмбеддингов)
  │
  └─→ Финальная версия (или повторный цикл редактор→критик, макс. 2 итерации)
```

### 4.3. Условие выхода из цикла

- Взвешенный скор Quality Radar ≥ 0.80 (порог настраиваемый).
- Максимум 2 итерации редактор→критик (защита от зацикливания и расхода токенов).
- Пустые сегменты (без текста) автоматически пропускаются.
- Пользователь может принять/отклонить на любом шаге.

## 5. Многовекторная оценка качества (Quality Radar)

### 5.1. Оси сравнения

| Ось | Метод | Реализация | Что измеряет |
|-----|-------|-----------|-------------|
| **Семантика** | API-эмбеддинги | OpenRouter (`all-MiniLM-L6-v2`) / ProxyAPI (`text-embedding-3-small`) | Смысловая близость |
| **Тональность** | LLM-оценка | Критик (`translation_critic`) — sentiment score 0–1 | Эмоциональный тон |
| **Ритмика** | Программная | `phoneticFeatures.ts` — syllable count, phrase length, BPM | Темп, длина фраз, ритм |
| **Фонетика** | Программная | `phoneticFeatures.ts` — consonant onset freqs, vowel stress, IPA-categories | Аллитерация, ассонанс, звуковой рисунок |
| **Культурный код** | LLM-оценка | Критик (`translation_critic`) — cultural score 0–1 | Идиомы, реалии, аллюзии |

### 5.2. Программная фонетика (F5)

**Принцип**: IPA как универсальный мост RU↔EN. «ш» и «sh» → `/ʃ/` → одна категория.

Файлы:
- `src/lib/phoneticFeatures.ts` — извлечение фонетических/ритмических фичей из текста
- `src/lib/embeddingClient.ts` — клиентский враппер для вызова семантических эмбеддингов
- `src/lib/qualityRadar.ts` — оркестратор: вычисляет все 5 осей, возвращает `RadarScores`

```
phoneticFeatures(text, lang) → PhoneticVector
├── syllableCount          ← ритмика
├── avgSyllableLength      ← ритмика
├── consonantOnsetFreqs[]  ← фонетика (аллитерация)
├── vowelStressFreqs[]     ← фонетика (ассонанс)
└── rhythmPattern          ← ритмика (ударные/безударные)
```

Формула взвешенного скора:
```
Score = Σ(wi · axiScore_i), где i ∈ {semantic, sentiment, rhythm, phonetic, cultural}
```
Веса настраиваются пользователем; пресеты: «Проза» (α semantic↑), «Поэзия» (β phonetic+rhythm↑).

### 5.3. UI: Радар качества перевода

- Пятиосевая радар-диаграмма (recharts) на уровне **сегмента** (клик по блоку в билингве).
- Цветовая индикация: зелёный (>0.85), жёлтый (0.70–0.85), красный (<0.70).
- Клик по оси → детализация: конкретные фразы с наибольшим отклонением.
- Двухуровневый **модульный кэш** (radarCache + computedCache) — мгновенное отображение при повторном выборе и навигации.
- Сохранённые оценки загружаются из `radar.json`; пересчёт — только для несохранённых сегментов.

### 5.4. Многослойная визуализация (F7b)

Радар отображает до 3 независимых слоёв, каждый со своими данными:

| Слой | Оси | Источник данных | Визуал |
|------|-----|----------------|--------|
| **3R (базис)** | Семантика, Ритмика, Фонетика | `radar-literal.json` — программный расчёт после подстрочника | Треугольник: тонкая сплошная линия `destructive` (strokeWidth 0.45), заливка 8% |
| **5R (арт-правка)** | Все 5 осей | `radar-literary.json` — полный 5R расчёт при арт-правке | Пятиугольник `primary`, recharts Radar |
| **5R+Alt (оценка)** | Все 5 осей | `radar-critique.json` — LLM-критика | Пятиугольник `accent`, recharts Radar |

**Принципы изоляции данных:**
- Каждый слой строго берёт данные из своего файла. Нет фолбэков между стадиями.
- 3R-треугольник рисуется через выделенный SVG-компонент `ThreeAxisRadarOverlay` поверх сетки (z-index: 20).
- 5R и 5R+Alt — стандартные recharts `<Radar>` с уникальными React-ключами и `isAnimationActive={false}`.
- Если на арт-правке 5R не рассчитался (ошибка критики) — пятиугольник не сохраняется, слой остаётся пустым.

**UI управления слоями:**
- Переключатели слоёв (3R всегда вкл, 5R/5R+Alt — toggle) и бейджи пресетов весов (Проза/Поэзия/Баланс) расположены в одном ряду в хедере монитора.
- Переключатели видимы всегда; недоступные стадии — с пониженной прозрачностью и `disabled`.
- Бейджи пресетов прибиты к правому краю через `ml-auto`.

**Компоненты:**
- `ThreeAxisRadarOverlay.tsx` — SVG-треугольник 3R (персистентный фон)
- `QualityRadarChart.tsx` — recharts RadarChart (5R + 5R+Alt слои, презентационный)
- `QualityMonitorPanel.tsx` — оркестратор: загрузка данных, управление слоями, пресеты весов

## 6. Батчинг и параллелизация (F8)

### 6.1. Три уровня перевода

| Уровень | Пайплайн | Вызов |
|---------|----------|-------|
| **Сегмент** | Только подстрочник (translate-literal) | Кнопка «Перевести» на сегменте |
| **Сцена** | Полный цикл (literal → literary → radar → critique × N итераций) | Кнопка «Полный пайплайн» |
| **Глава** | Все сцены главы параллельно через ModelPoolManager | Кнопка «Перевести главу» |

### 6.2. ModelPoolManager

- Round-robin распределение задач по пулу моделей (`art_translator` role).
- Ограниченная конкурентность (2 задачи/модель по умолчанию).
- Автоматический retry при 429/402, отключение модели после 3 ошибок.
- `TranslationProgressPanel` показывает прогресс, текущий этап пайплайна и статистику воркеров.

### 6.3. Файлы

- `src/hooks/useTranslationBatch.ts` — оркестратор батч-перевода (scene + chapter)
- `src/hooks/useSegmentTranslation.ts` — перевод отдельных сегментов (literal-only)
- `src/lib/translationPipeline.ts` — полный пайплайн per-segment
- `src/components/translation/TranslationProgressPanel.tsx` — UI прогресса

### 6.4. Edge Functions

| Функция | Назначение |
|---------|-----------|
| `translate-literal` | Батчевый подстрочный перевод (все сегменты сцены) |
| `translate-literary` | Художественная адаптация (per-segment, с контекстом) |
| `critique-translation` | LLM-оценка (sentiment + cultural + issues) |

### 6.5. Парсинг ответов AI

`translate-literal` парсит ответы AI с гибкой стратегией разбиения:
1. Маркеры `--- Segment N ---` (с/без trailing `---`)
2. Нумерованные строки (`1.`, `2)`)
3. Двойные переводы строк
4. Фоллбэк на один блок (для 1 сегмента)

## 7. Кэширование

### 7.1. Персистентный кэш (OPFS)

- `radar.json` — per-scene, содержит оценки всех сегментов, историю итераций, заметки критика.
- Записывается автоматически после полного пайплайна.

### 7.2. Модульный кэш (in-memory)

- `radarCache` — Map<sceneId, radar.json data> — предотвращает повторное чтение OPFS.
- `computedCache` — Map<sceneId:segmentId, { scores, notes }> — мгновенный отклик при навигации и ремаунте.
- Оба живут на уровне модуля, переживают ремаунт компонентов и переходы между страницами.

## 8. Ограничения и риски

- **Стоимость**: Heavy-tier модели для редактора и критика; эмбеддинги — дополнительные вызовы.
  Митигация: батчинг сегментов, кэширование эмбеддингов, лимит итераций.
- **Детерминизм**: LLM-перевод недетерминирован.
  Митигация: фиксация seed/temperature где возможно; версионирование результатов.
- **Поэзия/стихи**: Особый режим с приоритетом фонетической оси.
